## Remaining Gaps

- Review-mode parity is now partially landed on Android, but Remodex still has a richer review/revert surface, better branch-selection UX, and more complete review-result presentation.
- The public repo now includes a bundled generic relay push helper, but the actual Android delivery backend is still something you wire to your own provider or webhook.
- `thread/contextWindow/read` is now ported; `account-status` and voice helper RPCs remain unported.
- Remodex still has more onboarding flow depth and settings micro-interactions than Androdex.
- Some Android git UX remains less polished than Remodex even though the functional action set is now largely present.

## Intentional Platform Differences

- Host-local runtime stays the core product invariant. Androdex is not trying to move Codex execution onto the phone.
- macOS launchd service behavior is now treated as first-class, while Android remains the only mobile client.
- The public repo avoids hardcoded private relay domains, hosted credentials, and private push deployment runbooks.
- iOS-specific Remodex helpers still are not copied; Androdex keeps the macOS host pieces and Android-specific client code only.

## Manual Smoke Checklist

1. Run `androdex up`, pair from Android, and confirm saved reconnect works without rescanning.
2. Open one thread from Android and create another in a different project to confirm cross-project activation still switches host context correctly.
3. Start a run, verify streaming rows stay item-scoped, then press `Stop` before the turn publishes a usable `turnId` and confirm the fallback still interrupts correctly.
4. Background the app during an active run, reconnect or foreground it again, and confirm the active thread still shows the running state and the final completion lands in the existing timeline rows.
5. Resume a desktop-started thread from Android and confirm rollout live mirroring replays the pending user/assistant activity without duplicate fake thinking rows.
6. Use plan mode, queued follow-ups, `@file`, `$skill`, and `/subagents` in one thread and confirm queue restore and send behavior still work after reconnect.
7. Attach photos from gallery and camera, hit the 4-image cap, confirm loading/failure tiles behave correctly, and verify a restored queued draft keeps ready attachment previews.
8. Open the git sheet, load status and diff, create or switch a branch, create a managed worktree, and confirm safety prompts still prevent destructive branch/worktree flows.
9. With push configured, background the app, finish a run, tap the completion notification, and confirm Android reopens the target thread or shows the missing-thread prompt instead of dropping the route.

## Next Three Highest-Value Parity Tasks

1. Deepen Android review-mode parity: assistant revert UX, richer findings/results presentation, and better base-branch selection flows.
2. Close the remaining onboarding and trusted-pair UX gap: richer welcome/about presentation, clearer reconnect recovery, and stronger saved-pair status surfaces.
3. Decide the next bridge-helper scope explicitly, then port `account-status` and `voice` only if they still fit the host-local Android product.
