## Summary

Androdex already matches Remodex on the most important architectural invariant: Codex runs locally on the host, the phone is a paired remote client, transport is encrypted, workspace switching is remote-capable, and repo mutations stay host-local. The biggest parity gaps are now on the client state model and higher-level UX, not on the base relay/daemon shape.

The most important caution from `AGENTS.md` is that timeline correctness is still not at Remodex parity even though some item and turn identifiers are already present in the Android models. The current Android flow remains much closer to a simple thread reload model than to Remodex's item-aware reducer/service model.

## Prioritized gap list

1. Implement Remodex-style timeline/run-state semantics on Android: item-scoped reconciliation, per-thread active turn tracking, stop fallback via `thread/read`, late reasoning merge, reconnect/background recovery, and item-aware history reconciliation.
2. Upgrade the Android composer further with review flows and attachments on top of the now-matched slash/file/skill/runtime/fork affordances.
3. Port worktree-aware git bridge methods and build Android git UI for branch switching, status, diff, commit/push/pull, and worktree actions.
4. Add Android attachment support with camera/gallery intake, preview tiles, payload wiring, and attachment-state tests.
5. Add Android notification support end to end: bridge registration/completion hooks, mobile registration/open-thread handling, and foreground/background recovery.
6. Decide which Remodex bridge extras should come over next: notification handlers/live mirror first, then thread-context helpers, then voice/account only if they fit Androdex scope.
7. Expand onboarding/settings/about flows so trusted pairing, compatibility issues, reconnect recovery, and runtime state are easier to understand on Android.
8. Add regression coverage for the parity-sensitive areas above before larger refactors land.
